The Field: publish first, curate by pulling

A per-user namespace where anyone's uncurated use cases live and are linkable the day they're written — with a much lighter placement (just kind: lifecycle | enabling, plus an optional pointer). The curator stops being a gate at intake and becomes a pull: the Field is the pool the curated shelves and the featured strip draw from. This proposal lays out the IA options. Read 01–07, make the calls in 08, lock it at the bottom.

Where this sits in the stack

Builds on the two-spine model just implemented in PR #6 (lifecycle + enablement; one use-case entity; soft enabled_by): field notes may point at either spine, at any altitude. And it reshapes the submission loop the adjacent session is designing — intake becomes two paths (self-publish to your namespace, or an issue for drive-bys), and the curation queue starts reading the Field instead of only the issue tracker. §07 carries the updated coordination asks.

Why "the Field"

The site already uses the word. An empty activity page says "This activity is on the spine, but the field hasn't filled it in." The Practice voice is "what teams are doing" — field reports. So the uncurated surface is the Field, an entry is a field note, and promotion is elevation — vocabulary the playbook already speaks. (Rename freely in D6's notes if it doesn't sit right.)

01The premise

The gate is also a bottleneck — and a chill

Today every visible word passed the editor: a submission is an issue, and it becomes content only when the editor writes the file. That keeps the shelves impeccable — and caps throughput at one person's attention.

Today — the gate modelProposed — the pull model
A submission is…an issue, waiting in a queuecontent, live in your namespace the day you write it
Visible when…the editor elevates it (or never)immediately — unmistakably marked unreviewed
Placement burdeneditor does leaf-level placement at intakeauthor gives one word (kind) + an optional pointer; leaf placement happens at elevation
The curator is…a gate — nothing exists until they acta pull — the best of the Field gets elevated, voiced, placed
Author's stakecan't link to their own practicea durable, linkable record under their name
Curated promiseunchanged — the spines and the featured strip stay 100% editor-curated
  • Submissions die in queues. The half-written issue that never became a use case is practice the playbook lost. A namespace makes the default outcome "published rough," not "lost."
  • Authors need an address. "Look at my field note" is a different social contract than "I filed an issue" — it gives practice a URL, an owner, and pride of authorship before any editor touches it.
  • Curation gets better inputs. The curator browses a living pool — sortable, linkable, already in the file format — instead of reconstructing context from issue threads.

Precedents — this pattern has many names

Wiki user namespaces

MediaWiki's User: pages — personal space with full publishing rights, clearly outside the encyclopedia proper. Promotion = an editor moves content mainspace-ward.

Innersource / fork → upstream

Work freely in your fork; upstream pulls what clears the bar. Ownership by path; promotion by pull. The git-native shape of this proposal.

Draft → published workflows

Every CMS separates workflow state from content. The lesson: make state structural (where it lives), not just a flag someone forgets to check.

Digital gardens

Personal, perpetually rough notes published without apology — with the polish living elsewhere. The Field is the org's shared garden.

Scoped packages

npm's @user/pkg — a namespace that grants autonomy precisely because it can't collide with the curated registry's top level.

Answer first, curate later

Stack Overflow publishes everything instantly; curation (votes, accepted, dupes) is a lens applied after. Visibility and authority are different axes.
02The entity ladder

Field note → use case → featured / endorsed

One anatomy all the way up — Goal → Approach → Impact — so elevation is an edit, never a rewrite from scratch. What changes per rung is structure and authority, not shape.

The pull model — authorship flows right, provenance points back
What a rung grants — and what it forbids

Field note: same Goal → Approach → Impact body; frontmatter is just id, title, kind, optional near/tools/techniques. Owner derived from the path, never typed. Forbidden by validation: featured, endorsed, leaf placement fields — curation outputs can't be self-granted.

Elevation grants: leaf placement (activity or enabler), technique tags checked, the editor's note, eligibility for featured/endorsed — and the curated register of the Practice voice. The voices stay honest: a field note is raw practice; canon and the editor never speak in the Field.

03Storage

Where the namespace lives in the content store

The data-layer rule still governs — structured if you'd query it, Markdown if you'd read it — and adds one question: is "uncurated" a place or a flag?

ADirectory-per-user✦ recommended
content/field/<handle>/*.md — the namespace is the path; the loader reads the Field as a separate collection with owner derived from the directory.
content/
  usecases/              ← curated, two spines (unchanged)
  field/                 ← the Field (new)
    mreyes/
      f-triage-cache.md
      f-eval-notes.md
    okafor/
      f-dictionary-draft.md
Why it wins: uncurated is structural — no query can leak it into a curated view by forgetting a filter. Ownership is enforceable by path (CODEOWNERS / a one-line PR check: you may merge what's under your own handle). Deleting a namespace is rm -r. The curated pool's loader, validation, and pages don't change at all.
leak-proof by constructionpath-scoped merge rightscurated store untouchedone more collection in the loader

content/usecases/*.md grows status: field and owner: frontmatter; curated is status: curated (default).
one entity, one folderevery existing query needs a status filter — one miss leaks unreviewed content into the shelvesmerge rights can't be path-scopedowner typed, not derived
Verdict: workflow state as a flag is exactly the failure the CMS precedent warns about. Rejected unless the namespace idea itself shrinks to a handful of files.

04The light placement

One required word, one optional pointer

Curated use cases place at the leaf (activity / enabler), validated hard. A field note shouldn't have to know the taxonomy that well — but the more it says, the more surfaces can carry it.

content/field/mreyes/f-triage-cache.md — the whole schema
---
id: f-triage-cache
title: 'Caching system prompts across
        our triage fleet'
kind: lifecycle        # required: lifecycle | enabling
near: g-ops            # optional: ANY id on either spine —
                       # a phase, subphase, activity,
                       # track, area, or enabler
techniques: [t-context] # optional, validated if present
tools: [Claude Code]    # optional
---

system prompt every run…

## Approach
…

## Impact (optional)
…

Owner = the directory (@mreyes), never typed. No featured, endorsed, phase/activity, or track/enabler — validation rejects curation-only fields in the Field.

Why "near", not "placed"

near is deliberately humble: file this in the vicinity of X. It resolves against both spines at any tier (ids are globally unique across them — guaranteed by the PR #6 collision guards), validates only if present, and is replaced by real leaf placement at elevation. The author's guess never masquerades as the editor's filing.

05Surfacing

Where field notes appear on the site

The register must be unmistakable everywhere: dashed borders, a mono byline, an "unreviewed" microcopy — and never a seal, never the featured strip. Within that, four levels of presence.

ANamespace pages + contextual strips✦ recommended
Three surfaces: a Field index, per-user pages, and a quiet "From the field" strip on whichever spine page a note is near.
  • /field/ — the index: latest notes, by-author roll-up, an "unelevated, most linked-to" sort for the curator. The curation queue, in public.
  • /field/<handle>/ — a person's page: their notes, their elevated record ("3 notes elevated into the playbook").
  • /field/<handle>/<id>/ — the note itself: same Goal/Approach/Impact layout, dashed frame, byline, "Unreviewed field note" banner, and — if elevated — the "Elevated ↑" link to the curated version.
  • "From the field" strips — on phase/activity/track/enabler pages with notes near them: a compact, visually lighter row set below curated practice. This is what makes the one-word placement pay rent — and it finally gives empty activities a truthful state: "nothing curated yet, but two field notes nearby."

Navigation presence (D4 below)

If the Field gets surfaces, does it get a top-nav item? Lifecycle · Enablers · Techniques · Field · About says participation is first-class — at the cost of a fifth item. The quiet alternative: reach the Field from the submit flow, the footer, and the strips. Recommendation: top-level — the header already shouts "Submit a use case"; the Field is where that invitation finally leads to something the submitter owns.

06Elevation

The pull: how a field note becomes a use case

"The curator pulls in featured content to elevate" — mechanically, three ways to do the pull.

ACopy-on-elevate, provenance both ways✦ recommended
The curator copies the note into content/usecases/, edits freely (leaf placement, tags, trims, the editor's note), and records where it came from. The original stays the author's.
# content/usecases/u-triage-cache.md (curated copy)
---
id: u-triage-cache
title: 'Prompt caching across a triage fleet'
phase: grow
activity: g-ops
techniques: [t-context]
source: field/mreyes/f-triage-cache   # validated ref
author: Northstar Squad
---
# the field note needs NO new field —
# its "Elevated ↑" banner is the reverse join,
# derived at build time from `source`.
Why it wins: the curated version is an editorial artifact — voiced, placed, possibly cut in half — which is exactly what the editor already does to issue submissions. The author's original survives in their register and their namespace; the curator never edits inside someone else's directory; and the link is one validated frontmatter field with a derived reverse join (the same pattern as enabled_by). Double-counting is a non-issue: practice grids draw only from the curated pool, strips only from the Field — and an elevated note can drop out of the strips automatically.
author's record preservedcurator edits only curated spaceone new field, validatedtwo documents exist (by design)
BMove-on-elevate
git mv from the namespace into the curated pool; the user page lists "elevated" by link.
single instance, no driftthe author's original text is overwritten by editing — their record now lives only in git historynamespaces thin out exactly when their owners succeed
CFlag-in-place
The curator sets status: curated + leaf placement on the file where it sits; "curated" becomes a query, not a place.
curator edits inside user namespacesreintroduces the status-flag leak risk storage option A exists to preventcheapest single-file story
What the curator's queue looks like under A

The Field index, sorted for the editor: unelevated first, then by recency; later, by signal (how many curated pages a note sits near, GitHub reactions on its PR — additive, not structural). Elevation itself is one PR: add the curated file with source:, build-content validates the ref, the strips and banners update themselves. The gate didn't disappear — it moved to where it scales: after publication, on the editor's clock.

07Intake & the loop

Two ways in, one light gate

The submission loop stops being the only door and becomes the assisted one. Both paths end in the same place: a file in your namespace.

Path 1 — self-publish (git-native)

Open a PR that only touches content/field/<your-handle>/**. A one-line CI check verifies path-matches-author and runs the light validation; on green it can auto-merge. Your namespace, your merge — the org's innersource muscle, reused.

Path 2 — assisted (everyone else)

The existing issue templates stay, with one new outcome: instead of waiting for the editor to write a curated file, a bot (or the editor) scaffolds the field note into the submitter's namespace from the issue fields. Drive-by contributors get an address too.
The loop with the Field in it — publication first, curation as a pull

The light gate, precisely

  • Hard (build fails): duplicate id; missing id/title/kind; kind outside lifecycle | enabling; near / technique ids that don't resolve; any curation-only field (featured, endorsed, leaf placement, source) in the Field; a file outside the PR author's namespace.
  • Soft (warn only): missing ## Goal section; empty body. Rough is allowed — that's the point of the Field.
  • Curated side, one addition: source: must resolve to a real field note. Everything else about the curated gate is unchanged.
Coordination asks v2 — for the in-flight submission-loop session
  • Intake forks before the issue. The submit modal / CTA offers both doors: "publish to your Field namespace (PR)" and "open an issue, we'll scaffold it." Existing templates survive; their output target changes from "editor writes a curated file" to "a field note in your namespace."
  • The curation queue reads the Field, not just the tracker. Issues become an intake detail; the durable queue is "unelevated field notes," which the /field/ index already renders in public.
  • Status vocabulary gets a rung. field → elevated (curated) → featured / endorsed — shelf-agnostic, exactly one promotion path, the Field rung never wearing curation marks.

Worth saying out loud: field notes are public web pages, like issues are today — same publicness, better shape. Namespaces are GitHub handles (the merge check depends on it); team namespaces can come later as an alias map, a data edit. And nothing here touches PR #6's spines: the Field points at them but lives beside them.
