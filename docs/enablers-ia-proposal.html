AI PDLC Playbook · IA proposal · 2026-06-10

Enablers: new expectations, same loop

Some of the highest-leverage AI work isn't building a feature. It's platform owners, data owners, and product leaders making their corner of the org discoverable, machine-readable, and actionable — so everyone else's agents get traction. This proposal surveys the taxonomies the field already uses, drafts a flexible starting taxonomy, and lays out four IA options. Read 01–06, make the calls in 07, lock it at the bottom.

Relationship to the in-flight submission loop

An adjacent session is designing the submission/curation loop for lifecycle ideas. This proposal treats that loop as a shared rail: every option below rides it unchanged except for one fork — placement chooses a shelf. §06 lists the three properties that keep the loop shareable. Nothing here blocks on that work, and nothing in that work should need to be redone for this.

01The premise

A class of work the lifecycle can't hold

Every use case in the playbook today answers one question: how do I use agents to do my lifecycle work better? Enablers answer a different one: what do I owe everyone else, so that their agents work here?

Data owners

Make the data self-describing

  • Data dictionaries published machine-readable, not as PDFs or wiki prose
  • Registered in the catalog so agents (and their humans) can find them
  • Semantics, metrics, lineage and freshness exposed as structured signals
Platform owners

Make the platform agent-native

  • Platform context packs — architecture, processes, support model — discoverable and parseable
  • Interface specs current and complete; golden paths agents can actually follow
  • Agent identity, permissions and sandboxes as first-class access
Product leaders

Make direction addressable

  • Priorities and strategy at stable, citable locations — not trapped in decks
  • Principles and decision rules published so agents can apply them, not guess them
  • A durable decision log agents can ground on

Why these don't place on the lifecycle

  • They aren't phase-bound. A machine-readable data dictionary isn't a Discover, Build, or Grow act — it's continuous stewardship of an asset, alive across every effort that touches it. Any phase placement would be a misfile.
  • The beneficiary is inverted. A lifecycle use case improves the author's own throughput. An enabler's payoff lands downstream, in other teams' use cases — which is exactly why this work never finds room in the owner's feature backlog, and why the playbook should give it status.
  • Expectations need an address. "Here is what's expected of platform owners" has to be a shelf you can point someone at — owned, reviewable, endorsable — not a paragraph buried under Build › Documentation.
Working definitions

Enabler — a named, owned expectation: make asset X agent-ready, to standard Y. It carries the same three voices as everything else in the playbook: canon (the expectation itself), the editor's note, and practice.

Enabling use case — a worked example of meeting an enabler, with the same anatomy as any use case (Goal → Approach → Impact): "we published the dictionary as versioned YAML, registered it, wired freshness checks — downstream agent retrieval misses fell 40%."

Not a technique. A technique is how an approach works (subagents, evals, MCP). An enabler is a precondition you owe others. Techniques span both kinds of use case — one shared pool.

02What the field says

No settled canon — but four recurring dimensions

There is no agreed "enablement taxonomy" to import. There are eight serious attempts circling the same territory, and they disagree in instructive ways.

FrameworkShapeWhat we borrow
FAIR principles (2016)go-fair.org Four qualities of research data — Findable, Accessible, Interoperable, Reusable — with 15 sub-principles; explicitly "for machines as well as people." The original move: readiness expressed as qualities of an asset. Its vocabulary survives in every successor.
Data mesh · data-as-a-productmartinfowler.com (Dehghani) A data product must be discoverable, addressable, trustworthy, self-describing, interoperable, secure (DATSIS) — under federated computational governance. Expectations attach to an owned product, not a central function. Closest to the federated "you own it, you serve it" language the org already speaks.
Gartner — AI-ready datagartner.com Readiness = metadata management + data quality + observability; metadata moves "from passive documentation to active intelligence"; readiness is always relative to the use. The data-owner bucket's core canon: machine-navigable metadata, not human-only docs. BI-ready ≠ AI-ready.
CNCF — Platform Engineering Maturity Modeltag-app-delivery.cncf.io Five aspects — Investment, Adoption, Interfaces, Operations, Measurement — by four maturity levels; platform-as-a-product throughout. "Interfaces" is where agent-grade specs and golden paths live. The levels are the part we defer.
Microsoft — Agent Readiness Frameworklearn.microsoft.com Five pillars — Business & AI strategy, Process mapping, Technology & data, Org readiness & culture, Security & governance — by five maturity levels. Proof the field is converging on pillars × levels. Right shape for an executive roll-up later; too coarse for a playbook shelf.
Cloudflare — Agent Readiness score (2026)blog.cloudflare.com Five web-surface layers — Discoverability, Content, Bot access control, Capabilities, Commerce — scored by concrete checks: robots.txt, llms.txt, Markdown content negotiation, .well-known/api-catalog, MCP server card, OAuth discovery. The most use-case-shaped: readiness as checkable behaviors per surface. Every check has an internal analog — what's your platform's .well-known?
The AX movementagentexperience.ax · llms.txt · AGENTS.md Conventions, not a model: llms.txt reading lists, AGENTS.md operating manuals, MCP servers, OpenAPI-first design, errors an agent can self-correct from. The artifact layer. Many enabling use cases will literally be "we shipped the llms.txt / AGENTS.md / MCP card equivalent for our platform."
McKinsey — the agentic organizationmckinsey.com Six organizational shifts; named enablers: workforce upskilling, infrastructure adaptation, data productization, agent-specific governance. The leadership-bucket vocabulary — and confirmation that "enabler" is the field's own word for this work.

Adjacent and compatible: the semantic-layer / metadata-layer-for-AI work (including the multi-vendor Open Semantic Interchange spec) for machine-readable business definitions; data contracts as enforceable expectations; Diátaxis for structuring the prose humans still write.

The synthesis — four dimensions, three usable now

The steward · who

Anchor expectations to a role that owns the asset. Data mesh and McKinsey both land here. Use as the spine.

The surface · what

The asset being made agent-ready: data, interfaces, context, direction. Cloudflare's layers, Gartner's data, CNCF's interfaces. Use as the grouping tier.

The qualities · to what standard

FAIR and DATSIS distill what "ready" means, asset by asset. Use as shared vocabulary, not structure.

The maturity · how far

Levels and scores — the rigid part every big framework adds last. Defer until the taxonomy has settled.
The working definition this survey yields

An enabler is an owned asset, made agent-legible to a nameable standard, demonstrated by checkable behaviors.

Steward + surface + qualities give the taxonomy its three tiers. Maturity levels are real but premature — the seal already provides a binary rung (expected / not yet), which is all the rigidity the content can currently support.

The org context — no rigid taxonomy yet, by design

Unlike the PDLC, the org hasn't standardized an enablement taxonomy. What exists is softer: durable role buckets — platform owners, data owners, product leaders — and an established federated-data-ownership vocabulary. So the starting taxonomy anchors on those roles, keeps areas and enablers cheap to rename and regroup (the lifecycle.json machinery already proved renames are a data edit, not a redesign), and lets the real taxonomy emerge from what gets submitted.

03The starting taxonomy

Track › area › enabler — the spine shape, reused

Three tiers, mirroring phase › subphase › activity — so the compiler, the validator, the generated CSS, and the editor's mental model all transfer unchanged.

Track · data owners

Make your data legible to agents you'll never meet

Describe
  • Machine-readable data dictionary
  • Published semantics & metric definitions
Expose
  • Catalog registration & discoverability
  • Agent-grade access paths & contracts
Assure
  • Quality, lineage & freshness signals
Track · platform owners

Run the platform as a product whose newest user is an agent

Context
  • Platform context pack — architecture, processes, support model
  • Machine-actionable runbooks
Interfaces
  • Current, complete interface specs
  • Golden paths & scaffolds agents can follow
Access
  • Agent identity, permissions & sandboxes
Track · product leaders

Make direction something an agent can cite, not guess

Direction
  • Addressable priorities & strategy
  • Published principles & decision rules
Rationale
  • Durable decision log
  • Shared product glossary

A scaffold, not a standard — same status as the lifecycle's first-draft subphases. Roles ≈ surfaces at this stage (each role stewards a characteristic surface), which is what lets one tier carry both. New tracks (API owners, process owners, risk partners) are an array entry, not a redesign.

The quality vocabulary — what "agent-ready" means, asset by asset

Distilled from FAIR, DATSIS, and Cloudflare's checks. Used editorially — each enabler's canon line names the qualities it confers (see D6 for whether this ever becomes structured).

Discoverable
An agent — or its human — can find it without asking around.
Machine-readable
Parseable structure: schemas, YAML, Markdown. Not screenshot-shaped.
Addressable
A stable URL or id an agent can be pointed at — and cite.
Current
Updated inside the owner's definition of done, with freshness signals.
Permissioned
Safe agent access by default: identity, scopes, audit.

## Goal
Every team pointing an agent at our tables
re-derived what the columns meant — or worse, didn't.

## Approach
Published the dictionary as YAML in the owning repo,
registered it in the catalog, made updating it part
of the schema-change definition of done.

## Impact
Downstream agents cite the dictionary instead of
hallucinating semantics. Misses fell 40%.
Invariant — techniques stay one pool

Enabling use cases tag from the same techniques.json as lifecycle use cases — no parallel technique list. t-mcp, t-structured, and t-retrieval are the obvious workhorses on this side, and a technique page simply aggregates practice from both axes. This is the "same IA layer" the owner asked for — and it's what makes the two halves feel like one playbook.

04The IA options

Four ways to shelve the same content

All four keep the invariants: one use-case anatomy (Goal → Approach → Impact), one technique pool, one submission loop, one endorsement seal, one build gate. They differ only in how much structure enablers themselves get.

Enablers become a peer browse axis: enablers.json beside lifecycle.json, compiled, validated, and rendered by the same machinery.

Recommendation — a composite, not a winner-take-all

Take A, fold in C, write B, defer D. Ship the second spine with C's enabled_by edge as part of it; write the track pages in B's voice (that prose is needed anyway); revisit D once enough enablers carry the seal that levels would mean something. The four options were never rivals — B is A's prose, C is A's edges, D is A's executive view, later.

The join graph — two spines, one entity model, one technique pool

What each page gains

The enabler page

Four blocks: the expectation (canon), the editor's note, practice (enabling use cases placed here), and unlocks — every lifecycle use case that declares it in enabled_by. The unlocks list is the demand evidence.

The use-case page

The sidebar grows an "Enabled by" chip row beside techniques and tools — each chip linking to the enabler it leans on. The owner's sentence, rendered.

The technique page

Now aggregates practice from both spines — one #structured-output page shows the lifecycle use cases and the enabling use cases that use it. No change to the technique model.
A curation signal, free of charge

enabled_by counts tell the editor which expectations the org is actually leaning on. An enabler that many lifecycle use cases cite is earning its seal; an enabler nothing cites is either ahead of its time or wrongly framed. Endorsement stops being a taste call and starts being evidenced.

06The shared loop

One loop, two shelves

The submission/curation loop being built for lifecycle ideas needs exactly one new fork to carry enablers: placement chooses a shelf. Everything else — intake, triage, voices, the seal, the build gate — is shared rail.

The loop — the fork at "place it" is the only new step
Three asks of the in-flight loop work — what keeps the rail shareable
  • Placement is a curation step, not a submission field. Submitters describe the work; the editor decides the shelf. Intake stays spine-agnostic.
  • Placement targets come from a registry, not a hardcoded list. Anything id-keyed and compiled (today lifecycle.json, tomorrow enablers.json) is a valid shelf.
  • Status and endorsement are shelf-agnostic. One label vocabulary (use-case, enabler, …), one promotion path to the seal.

The intake template — a sketch

# .github/ISSUE_TEMPLATE/enabler.yml — abbreviated
name: New enabler / enabling use case
description: Propose an expectation that makes the org
  easier to build in — or share how you met one.
labels: ["enabler"]
body:
  - Expectation      # one line — what should be true:
                     # "our data dictionary is machine-
                     #  readable and registered"
  - Who owes it      # platform owner · data owner ·
                     # product leader · other
  - The asset        # the dataset / platform / process /
                     # doc being made agent-ready
  - Why now          # what broke, or what it unlocks
  - How you met it   # if practice: approach + links
                     # to the artifact itself
  - Unlocks          # lifecycle use cases it accelerates
  - Techniques       # same shared tags

Mirrors use-case.yml's shape — title, substance, optional placement hint, tags — with two deliberate differences: it asks for the asset and the owing role (the two anchors of the taxonomy), and it asks what the work unlocks, which seeds the enabled_by edges at intake time.

One template can serve both a proposed expectation (no "how you met it" yet) and a practice report (all fields). The editor's triage separates them — exactly as it separates a use case from a technique suggestion today.

What the seal means over here

The brass seal already carries promotion semantics: proven practice, adopted into the protocol's recommendations. On the enablement spine the same seal reads one notch stronger: an endorsed enabler is a formal expectation of the role — "the protocol now expects platform owners to do this." Same mark, same promotion motion, same curation bar; the shelf supplies the strength of the verb. (That's D5 — including whether "expectation" deserves a distinct visual status instead.)

07Your call

Six decisions

Pre-set to the recommendation. Override anything, leave notes where you want to steer, then lock it at the bottom.

D1Which IA do enablers get?
The §04 options. This is the structural call everything else hangs off.
D2What names the top tier?
The §02 synthesis: steward, surface, or quality could each anchor the spine. The org's durable buckets are roles.
D3Is an enabling use case the same entity?
The owner's instinct: "enabling use cases are likely analogous to lifecycle use cases."
D4How do lifecycle use cases reference enablers?
The "use case × technique, enabled by [a, b, c]" linkage from §05.
D5What does the seal mean on an enabler?
On the lifecycle, the brass seal = "adopted into the protocol's recommendations." Enablers are expectations, so the seal reads stronger.
D6The quality vocabulary — editorial or structured?
Discoverable · machine-readable · addressable · current · permissioned (§03). Structured-if-you'd-query-it says: not until something queries it.
±Overall verdict & anything else
Sequencing, scope cuts, naming ("Enablers" vs "Readiness" vs "Expectations" in the nav), seed content priorities, coordination notes for the submission-loop session.
